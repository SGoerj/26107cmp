## 1. 项目背景

### 我们要解决什么问题？

在真实的渗透测试中，攻击者需要做出一系列决策：

- "先侦察哪个目标？"
- "这个服务器有什么漏洞？"
- "用 CVE 漏洞利用还是弱口令爆破？"
- "拿到权限后，要不要提权？"
- "通过哪个跳板横向移动到下一个目标？"
- "什么时候窃取数据撤退？"

每一步都有风险——被入侵检测系统发现、触发告警、留下痕迹。

**问题**：攻击者如何在一个复杂网络中，自主选择最优的攻击序列，在"到达目标"和"避免被发现"之间取得平衡？

### 我们的方法

用**深度强化学习**训练一个智能体，让它在一个模拟的企业内网环境中，自主决策完整的攻击链。

> **深度强化学习（Deep Reinforcement Learning, DRL）**：让 AI 通过"试错"来学习最优策略。智能体在环境中执行动作，获得奖励或惩罚，逐步学会在什么状态下应该做什么。

## 2. 从 v1 到 v2 的演进

### v1：只会走路（路径选择）

v1 的智能体只会做一件事——选择移动到哪个节点：

```
Internet → VPN → Jump → DB-Master   （安全路径）
Internet → Web → DB-Master           （危险路径，被检测风险高）
```

这是一个**路径选择**问题。网络中的节点是同质的——没有服务、没有漏洞、不需要攻击。DQN 只需要学会挑哪条路。

> **DQN（Deep Q-Network）**：用神经网络来近似 Q 函数——一个从"状态→动作"到"期望奖励"的映射。智能体选择 Q 值最高的动作。

### v2：会走路 + 会攻击（攻击链决策）

v2 的智能体除了会移动，还要学会：

| 阶段 | 动作 | 含义 |
|------|------|------|
| 🔍 侦察 | RECON | 扫描当前节点运行了哪些服务（SSH？HTTP？MySQL？） |
| 🔬 漏洞评估 | VULN_SCAN | 扫描已知服务中是否存在漏洞（CVE？弱口令？SQL 注入？） |
| 💥 漏洞利用 | EXPLOIT | 根据发现的漏洞类型，使用对应的攻击手段获取访问权 |
| ⬆️ 权限提升 | PRIVESC | 从普通用户级别提升到 root/管理员级别 |
| ➡️ 横向移动 | MOVE | 从已控制的节点跳转到相邻节点 |
| 🏁 数据窃取 | EXFIL | 在 root 权限下窃取目标数据（flag） |

每一步都有严格的前置条件——你不能对还没侦察过的节点发起漏洞利用，也不能在还没拿到 root 权限时窃取数据。这迫使智能体学会**正确的攻击序列**。

## 3. 环境设计：网络拓扑

### 4 节点企业内网模拟

```
               [0] Gateway (检测率=0.05)
               (SSH, HTTP) — 已公开, 攻击者已 root
              /    \
             /      \
   [1] WebServer    [2] JumpBox
   (HTTP)           (SSH)
   检测率=0.70       检测率=0.15
   漏洞: CVE-2021-41773  漏洞: 弱口令
        |                |
        └──── [3] DC ★ ───┘
        (DB)
        检测率=0.90
        漏洞: CVE
        ★ FLAG: classified_data.tar
```

> **检测率（Detection Probability）**：节点上的 IDS/IPS 系统检测到攻击行为的概率。范围 0~1。Gateway 的 0.05 说明只有 5% 概率被检测到，DC 的 0.90 说明几乎每次操作都可能被发现。

> **CVE（Common Vulnerabilities and Exposures）**：公开已知的安全漏洞编号。例如 CVE-2021-41773 是 Apache HTTP Server 的路径穿越漏洞。

> **弱口令（Weak Credentials）**：使用简单密码（如 admin/admin123）或默认密码，攻击者可以通过暴力破解或字典攻击获取访问权。

> **SQL 注入（SQL Injection, SQLI）**：通过在 Web 应用的输入中嵌入恶意 SQL 代码，操控后端数据库的攻击方式。

### 为什么选 4 个节点？

v2 最初设计是 6 个节点，但智能体无法学会完整攻击链。原因：

- **组合爆炸**：14 步 × 13 种动作，纯随机探索几乎不可能偶然找到 flag
- **稀疏奖励**：只有最终窃取 flag 才有大奖励，中间步骤的奖励太微弱
- **目标恐惧**：DC 检测率 0.85 太高，智能体学会了"远离 DC 比完成任务更安全"

4 节点版本保持了核心学习目标（**选 Web 还是 JumpBox？**），同时把攻击链从 14 步缩短到 10 步，让 DQN 能够在合理时间内收敛。

> **稀疏奖励（Sparse Reward）**：只有完成最终目标才获得大奖励，中间没有足够多的反馈信号来引导智能体。这是强化学习中最常见的困难之一。

## 4. 攻击链：从侦察到数据窃取

### 完整攻击序列（教科书版本）

以安全路径（JumpBox）为例：

| 步 | 动作 | 当前节点 | 发生了什么 |
|----|------|---------|-----------|
| 1 | MOVE to JumpBox | Gate → Jump | 从已控制的 Gateway 跳转到 JumpBox（检测率只有 0.15，很安全） |
| 2 | RECON | JumpBox | 扫描发现 JumpBox 运行 SSH 服务 |
| 3 | VULN_SCAN | JumpBox | 扫描发现 SSH 存在弱口令漏洞 |
| 4 | EXPLOIT_CREDS | JumpBox | 用弱口令攻击成功获得 JumpBox 访问权 |
| 5 | MOVE to DC | Jump → DC | 通过 JumpBox 跳转到域控制器 DC |
| 6 | RECON | DC | 扫描发现 DC 运行 DB 服务 |
| 7 | VULN_SCAN | DC | 扫描发现 DC 存在 CVE 漏洞 |
| 8 | EXPLOIT_CVE | DC | 用 CVE 漏洞利用获得 DC 访问权 |
| 9 | PRIVESC | DC | 权限提升：user → root |
| 10 | EXFIL | DC | 🏁 窃取 classified_data.tar！ |

> **横向移动（Lateral Movement）**：攻击者在控制一台机器后，通过网络中的连接关系跳转到相邻机器的技术。JumpBox（跳板机）是攻击者经常利用的中间节点。

> **权限提升（Privilege Escalation, PRIVESC）**：从低权限用户（如普通域用户）提升到高权限用户（root/Administrator/SYSTEM）的过程。许多敏感操作（如读取特定文件）需要 root 权限。

> **数据窃取（Exfiltration, EXFIL）**：将目标数据从受害网络中提取出来的最终动作。在真实攻击中通常涉及加密传输、DNS 隧道等隐蔽通道技术。

### 两条路径的安全对比

| | 快路径（WebServer） | 安全路径（JumpBox） |
|---|---|---|
| 入口节点 | WebServer (检测 0.70) | JumpBox (检测 0.15) |
| 入口漏洞 | CVE | 弱口令 |
| 平均被检测次数 | ~3.2 | ~2.6 |
| 攻击链长度 | 10 步 | 10 步 |

智能体需要学会：**优先选 JumpBox，因为它检测率低，整体风险更小。**

## 5. 动作空间：智能体能做什么

智能体在每个时间步可以从 11 种动作中选一个：

| 动作 | 编号 | 说明 | 前置条件 | 被检测风险 |
|------|------|------|---------|-----------|
| RECON | 0 | 扫描当前节点运行的服务 | 未扫描过 | 0.1 × 节点检测率 |
| VULN_SCAN | 1 | 扫描已知服务的漏洞 | 已完成 RECON | 0.1 × 节点检测率 |
| EXPLOIT_CVE | 2 | CVE 漏洞利用 | 已完成 VULN_SCAN + 该节点存在 CVE 漏洞 | 0.5 × 节点检测率 |
| EXPLOIT_CREDS | 3 | 弱口令/凭据攻击 | 已完成 VULN_SCAN + 该节点存在弱口令漏洞 | 0.5 × 节点检测率 |
| EXPLOIT_SQLI | 4 | SQL 注入攻击 | 已完成 VULN_SCAN + 该节点存在 SQLI 漏洞 | 0.5 × 节点检测率 |
| PRIVESC | 5 | 权限提升 (user → root) | 已攻陷当前节点，尚未 root | 1.0 × 节点检测率 |
| EXFIL | 6 | 窃取 flag 数据 | 已 root + 当前节点有 flag | 1.0 × 节点检测率 |
| MOVE_0~3 | 7-10 | 横向移动到目标节点 | 目标节点与当前节点邻接 + 已攻陷当前节点 | 0.3 × 节点检测率 |

> **动作掩码（Action Masking）**：每一步只允许合法动作被选中。例如，智能体不能在同一节点重复做 RECON，也不能对没有 flag 的节点尝试 EXFIL。这大大缩小了搜索空间。

## 6. 状态表示：智能体知道什么

智能体在每个时间步观测到一个 **27 维向量**，编码了它对当前环境的**可观测**认知（POMDP 部分可观测）：

| 维度 | 内容 | 示例值 |
|------|------|--------|
| [0:4] | 当前在哪个节点（one-hot） | `[1,0,0,0]` = 在 Gateway |
| [4:8] | 哪些节点已被攻陷（自己做过，可见） | `[1,0,0,0]` = 只有 Gateway 被攻陷 |
| [8:12] | 哪些节点已完成 RECON | `[1,1,0,0]` = Gate 和 Web 已完成侦察 |
| [12:16] | 哪些节点已完成 VULN_SCAN | `[1,1,0,0]` = 同上 |
| [16:19] | 当前节点已知运行哪些服务 (SSH/HTTP/DB) | `[0,1,0]` = 只有 HTTP |
| [19:22] | 当前节点已知存在哪些漏洞 (CVE/Creds/SQLI) | `[1,0,0]` = 只有 CVE |
| [22] | 已用步数（归一化） | `0.25` = 走了 10 步（max 40） |
| [23:27] | 哪些节点曾经到访过（防振荡） | `[1,1,0,0]` = 去过 Gate 和 Web |

> **One-hot 编码**：用只有一个位置是 1、其余是 0 的向量表示离散值。例如 4 个节点用 `[1,0,0,0]`、`[0,1,0,0]` 等编码。

关键设计：
- **逐步揭示信息**：智能体只有在完成 RECON 后才能知道服务，完成 VULN_SCAN 后才能知道漏洞。这模拟真实攻击中的"情报收集"过程。
- **visited 记录**：防止智能体在两个节点之间无限振荡（Gateway ↔ JumpBox 来回跳）。
- **隐藏检测状态（POMDP 第一步）**：已移除"累计被检测次数"这一位。真实渗透中攻击者**看不到** IDS 后台的告警计数，只能从"工具被拦、连接被断"等**后果**推断暴露。智能体同样只能从检测**惩罚奖励**（r 里的 -5）间接学到"这个节点危险"，而非从仪表盘读数。这是向真实推进的关键一步。

## 7. 奖赏设计：如何引导学习

每执行一个动作，环境返回一个奖赏值。智能体的目标是最大化累计奖赏。

| 事件 | 奖赏 | 设计意图 |
|------|------|---------|
| 每步消耗 | -0.3 | 鼓励快速完成任务，少走弯路 |
| RECON 成功 | +0.5 | 给予情报收集正向反馈 |
| VULN_SCAN 成功 | +0.5 | 同上 |
| EXPLOIT 成功 | +3.0 | 重要里程碑——获得了节点控制权 |
| EXPLOIT 失败 | -2.0 | 惩罚不匹配的漏洞利用（已被动作掩码消除） |
| PRIVESC 成功 | +5.0 | 获得 root 权限是高价值动作 |
| EXFIL 成功 | +25.0 | 最终目标——最大的奖励 |
| 被 IDS 检测到 | -5.0 | 惩罚高风险行为 |
| 进入新节点 | +2.0 | 探索奖励——鼓励发现新路径 |
| 回到已访问节点 | -2.0 | 防止在两个节点间振荡 |

> **奖赏塑形（Reward Shaping）**：通过精心设计中间步骤的奖励（不仅是最终目标），引导智能体逐步接近目标。例如 +0.5 的 RECON 奖励告诉智能体"先侦察是正确的"。

> **探索奖励（Exploration Bonus）**：当智能体到达一个从未去过的节点时获得额外奖励。这解决了"智能体一开始不知道该去哪"的问题。

## 8. Double Dueling DQN 算法

### 为什么不用普通 DQN？

普通 DQN 有一个已知缺陷：**Q 值高估**（Overestimation Bias）。用一个网络同时选择最优动作和评估其价值，会导致 Q 值被系统性高估。

### Double DQN

**Double DQN** 用两个网络解耦动作选择和价值评估：

1. **策略网络（Policy Network）**：选择最优动作 `a* = argmax Q_policy(s')`
2. **目标网络（Target Network）**：评估该动作的价值 `Q_target(s', a*)`

Target = `r + γ × Q_target(s', argmax Q_policy(s'))`

> **Q 值**：在状态 s 执行动作 a 后，未来可以获得的所有奖励的折现总和。Q(s,a) 越高 = 这个动作越好。

> **γ（折扣因子, Gamma）**：0~1 之间。γ=0.99 表示未来的奖励几乎和当下的奖励一样重要；γ=0 表示只关心立刻的奖励。

### Dueling Network

**Dueling Network** 把 Q 值拆成两个独立的部分：

```
Q(s, a) = V(s) + A(s, a) - mean(A(s, :))
```

- **V(s)**：状态价值——"站在这个节点上，整体局面有多好？"（跟具体动作无关）
- **A(s, a)**：动作优势——"相比于平均水平，这个动作好多少？"

为什么这更好？因为有些状态下，做什么动作都不太重要（比如已经 root 了 DC，区别不大），而另一些状态下，动作的选择至关重要（比如在 Gateway 决定走 Web 还是 JumpBox）。Dueling 结构可以更高效地学习这种差异。

```
网络结构:
  27D 输入
   ↓
  FC(128) + ReLU
   ↓
  FC(64) + ReLU
   ↓
  ┌──────────────┐
  │              │
  V(s)          A(s,a)
  FC(32)+ReLU   FC(32)+ReLU
  FC(1)         FC(11)
  ↓              ↓
  └────── Q(s,a) ────┘
```

> **FC（全连接层, Fully Connected）**：神经网络中最基本的层，每个输入节点都与每个输出节点相连。

> **ReLU（Rectified Linear Unit）**：最常用的激活函数。`ReLU(x) = max(0, x)`，给网络引入非线性能力。

## 9. 训练过程与收敛分析

### 超参数

| 参数 | 值 | 含义 |
|------|-----|------|
| 训练轮数 | 1500 | 智能体从零开始尝试 1500 次完整攻击 |
| 批次大小 | 128 | 每次从经验池中抽取 128 条经验来更新网络 |
| 学习率 | 0.0005 | 网络权重每次更新的步长 |
| γ (折扣因子) | 0.99 | 未来的奖励几乎和现在一样重要 |
| ε 起始值 | 1.0 | 一开始 100% 随机探索 |
| ε 最终值 | 0.05 | 训练结束时只有 5% 随机探索 |
| ε 衰减率 | 0.998 | 每轮后 ε × 0.998 |
| 目标网络更新 | 每 25 轮 | 每 25 轮把策略网络参数同步到目标网络 |

> **ε-greedy 探索**：以概率 ε 随机选择一个动作（探索），以概率 1-ε 选择当前认为最好的动作（利用）。随着训练进行，ε 逐渐降低——从"随便试试"过渡到"相信自己的判断"。

> **经验回放（Experience Replay）**：把过去的经验 (s, a, r, s', done) 存在一个缓冲区里，训练时随机抽取一批。打破了经验之间的时间相关性，让学习更稳定。

> **目标网络（Target Network）**：一个定期从策略网络拷贝参数的"副本"。因为 Double DQN 用两个网络分别选动作和估值，目标网络提供了一个稳定的估值基准。

### 训练曲线

```
Ep  100: avg100 = -34.92, flag% = 65%   ← 高随机性，偶尔撞到 flag
Ep  300: avg100 =  +2.65, flag% = 100%  ← 首次稳定收敛！
Ep  500: avg100 = +11.21, flag% = 100%  ← 策略持续优化
Ep 1000: avg100 = +18.11, flag% = 100%  ← 接近最佳
Ep 1500: avg100 = +20.66, flag% = 100%  ← 最终收敛
```

**avg100**：最近 100 轮的平均奖励，用于平滑评估训练趋势。

**flag%**：最近 100 轮中成功窃取 flag 的比例。Ep 300 开始稳定在 100%，意味着智能体学会了可靠地完成任务。

### 训练耗时

- 硬件：anode16（CPU 节点，2 核）
- 1500 轮 × ~0.1 秒/轮 ≈ **2.5 分钟**
- 网络很小（28→128→64→11），CPU 训练完全够用

### 训练日志中的关键指标

```
[Ep   300/1500] reward= -26.10  avg100=   2.65  ε=0.548  flag%=100%  det=5 ★
```

| 字段 | 含义 |
|------|------|
| Ep 300/1500 | 当前第 300 轮，共 1500 轮 |
| reward = -26.10 | 这一轮的净奖励（被检测扣了分） |
| avg100 = 2.65 | 近 100 轮平均奖励——正值说明平均能获利 |
| ε = 0.548 | 还有 54.8% 概率随机探索 |
| flag% = 100% | 近 100 轮全部成功拿到 flag |
| det = 5 | 这一轮被检测到 5 次 |
| ★ | 新的最高 avg100 记录 |

## 10. 从 DQN 到 PPO：一次必要的换轨

第 8、9 章描述的 Double Dueling DQN 是这个项目的第一代算法。经过多轮调参和实验，我们用实证发现 **DQN 在本环境"学不深"**，最终换用 PPO。这一章记录这个转折——它是项目里最重要的一课。

### 10.1 DQN 的症状：训练后期策略持续退化

DQN 的训练曲线呈现一个固定的病态模式：

```
Ep  100  avg100= +10   flag%=95%   ← 高探索期 (ε≈0.9), 随机动作偶尔撞 flag
Ep  400  avg100=  -4   flag%=77%
Ep  900  avg100= -33   flag%=42%   ← ε 降下来, 真策略暴露, 一路下滑
Ep 2000  avg100= -52   flag%= 7%   ← 训练结束, 策略已严重退化
```

**关键观察**：只有 ε>0.6 时 avg100 才是正的。这是因为 60%+ 的随机动作偶尔会碰巧走通攻击链拿 flag，拉高了 avg100。一旦 ε 降下来、智能体真按自己学到的策略走，就振荡——后期 flag% 跌到 7%。

### 10.2 四次实验指向同一结论

| 实验 | 改动 | 结果 |
|------|------|------|
| 原版 DQN | 指数衰减 ε + best 快照 | 演示有时成功 (best 抓高探索期) |
| PER + 固定 ε | 优先回放 + ε=0.20 | 更糟 (失败经验被集中放大) |
| 严格 best | 只在 ε≤0.3 后记录 best | best 为负 (低探索期没好策略) |
| 宽松 best | 任何时候最高 avg100 | best 抓高探索期, greedy 演示振荡 |

四次实验揭示了一个之前没挑明的真相：**之前所有"成功演示"都是 best 快照在高探索期碰巧抓到能走通的策略，不是 DQN 真正学到了稳定的好策略。** 一旦严格要求"低探索期学好的策略"，就抓不到了——因为低探索期策略一直是负的。

### 10.3 根因：经验回放 + 灾难性遗忘

DQN 是 **off-policy**：把所有经验 (s,a,r,s',done) 存进回放池，训练时随机抽取。问题在于：

- 高探索期产生大量"碰巧走通"的经验（带正奖励），Q 网络学到这些
- ε 衰减后，智能体按学到的策略走，产生大量振荡/回访经验（带负奖励）
- 这些坏经验**覆盖**了好经验，Q 网络拟合漂移，把好策略"忘掉"了
- 越训练越差，形成恶性循环

这是 value-based off-policy 方法的结构性问题，不是调参能救的。PER（优先回放大误差经验）反而加速恶化——因为失败经验的 TD 误差最大，被集中回放。

### 10.4 解法：换 PPO（on-policy）

**PPO (Proximal Policy Optimization)** 从根本上避免了遗忘，因为它是 **on-policy**：

| | DQN (off-policy) | PPO (on-policy) |
|---|---|---|
| 数据来源 | 经验回放池（旧经验会被污染） | 当前策略实时采样，用完即弃 |
| 遗忘 | 严重——好策略被坏经验覆盖 | **天然免疫**——不用旧数据 |
| 学什么 | Q 值（贪心提取不稳定） | 策略概率 π(a\|s)（直接优化） |
| 探索 | ε-greedy（纯随机噪声） | 熵正则（有方向的探索） |
| 评估 | 需要 best 快照兜底 | **final 即 best** |

PPO 的核心组件：
- **Actor-Critic 网络**：共享底层特征，actor 头输出动作 logits，critic 头估状态价值 V(s)
- **GAE (Generalized Advantage Estimation)**：算优势函数 A_t，平衡偏差与方差
- **Clip surrogate loss**：限制新旧策略比例在 [1-ε, 1+ε]，防止单次更新步子太大
- **熵正则**：鼓励策略保持一定随机性，替代 ε-greedy 做探索
- **Action Masking**：非法动作 logits 置 -inf，softmax 后概率为 0（继承 DQN 版本的合法动作逻辑）

### 10.5 PPO 的结果：质变

同样的 4 节点环境、同样的 27D POMDP 状态，PPO 训练曲线和 DQN 完全不同：

```
Rollout  1  avg= 14.37  flag%= 96%
Rollout  5  avg= 29.41  flag%=100%
Rollout 30  avg= 30.66  flag%=100%
Rollout 60  avg= 32.34  flag%=100%   ← 训练结束, 稳定不退化
```

- **训练趋势**：持续上升并稳定在 +32，**没有 DQN 那种"越学越差"**
- **后期 flag%**：稳定 100%（DQN 跌到 7%）
- **演示用 final**（不是 best 快照）：greedy 11 步拿 flag——说明学到的是真实稳定策略
- **entropy**：1.73 → 0.69，策略从均匀探索收敛到确定动作，符合预期

这一步证明：环境建模（拓扑外部化、POMDP 隐藏检测）都没问题，问题在算法。换对算法后，同样的环境就能稳定学好。

### 10.6 工程教训

1. **评估 RL 策略要看"低探索期的真实表现"，不是高探索期的 avg**。高 ε 下的高 avg 是探索噪声，不代表策略质量。
2. **off-policy + 经验回放在攻击链这类环境容易遗忘**。路径有顺序、好经验稀缺，回放池污染后果严重。on-policy 更稳。
3. **算法选择比调参重要**。DQN 调了四轮（PER、固定 ε、best 逻辑）都没解决，换 PPO 一次到位。先确认算法适合问题，再调参。
4. **换算法时环境不动**。我们积累的环境外部化（YAML）、POMDP 状态设计都复用了——只换 agent，不重做环境。

## 11. 动态检测：让环境真正"动"起来

前面所有改进（外部化、POMDP、加节点、换 PPO）都在改"环境结构"。这一步改的是**环境行为**——风险会随智能体的动作变化，这是真实攻防和静态环境的关键区别。

### 11.1 动机：静态检测不够真实

之前每个节点的检测率是**固定常数**（如 JumpBox 恒为 0.15）。智能体学会"JumpBox 比 WebServer 安全"就一劳永逸。但真实 IDS 不是这样：

- 攻击行为触发告警 → SOC 警觉 → 该节点的检测能力**升级**
- 确认入侵 → 重点监控 → 检测率进一步上升
- 长时间无异常 → 警觉度**衰减**回落

也就是说，**风险是智能体自己"打"出来的**。在一个节点上反复攻击，等于把自己的暴露面越扩越大。这制造了静态环境没有的张力："快"（多打几下发起新攻击）vs "隐蔽"（打完就走，别在一处停留）。

### 11.2 规则式动态检测

每个节点维护一个**动态检测率** `det_level`，从 base 出发，按规则升降：

| 事件 | 检测率变化 | 含义 |
|------|-----------|------|
| 攻击性动作 (EXPLOIT/PRIVESC/EXFIL) | `+ α_attack` (0.08) | 攻击触发告警，SOC 警觉 |
| 被检测到 1 次 | `+ α_detected` (0.15) | 确认入侵，重点监控 |
| 每个 step（无新攻击） | `− α_decay` (0.01) | 没动静，警觉度自然衰减 |
| clip 到 | `[base, cap_high=0.95]` | 不低于基线，不高于 95% |

配置在 `configs/env_default_dynamic.yaml`，拓扑与静态版完全相同，仅开启动态检测。`env_v2` 里 `_update_dynamic_detection()` 在每个 step 更新当前节点的 `det_level`，`_roll_detection` 用动态值而非 base。`enabled=False` 时 det_level 恒为 base，行为与静态环境完全一致（向后兼容）。

### 11.3 POMDP 设计：检测率不可见

动态检测率**不进入状态向量**。智能体看不到"这个节点现在多危险"，只能从**检测惩罚奖励 (−5)** 间接推断"这地方被我搞警觉了"。

这延续了第 6 章 POMDP A 步的设计哲学（隐藏被检测次数）：能从奖励信号学的信息，就不该免费暴露给状态。真实渗透者也看不到 IDS 后台的警觉度，只能从"工具被拦、连接被断"的后果感受。

这是有风险的选择——智能体可能学不会"检测率涨了要走"。但 PPO 的实验证明它学到了（见 11.4）。如果学不会，退路是把检测率加回状态（变成"透明动态检测"），但张力会减半。

### 11.4 PPO 学到了什么："打了就跑"

日志 `rl-agent-v2-21220.out`，PPO 在动态环境下稳定收敛（avg +30、flag% 100%），演示展示了静态环境学不到的行为：

```
0  MOVE_2   Gateway → JumpBox
1  RECON    JumpBox
2  VULN_SCAN JumpBox  ⚠️ 被检测    ← JumpBox 开始被警觉
3  EXPLOIT_CREDS  ✅ 攻陷
4  PRIVESC  JumpBox
5  MOVE_3   JumpBox → DC           ← 关键: 攻陷+提权后立即离开, 不在 JumpBox 停留
6-10  DC 上完成攻击链, EXFIL 🏁
```

对比静态环境（21109）的演示：静态版在 JumpBox 攻陷后也是立刻走，但那是因为"没理由留"。动态版是"**不敢留**"——继续在 JumpBox 动作会让它的检测率从 0.15 飙到 0.5+，后续每步都可能被抓。PPO 学会了这个权衡。

被检测分布也印证：3 次检测分散在 JumpBox (2次) 和 DC (1次)，没有在一个节点上被反复抓。智能体接受了"不可避免的少量检测"，但避免了"在一处累积风险"的灾难。

### 11.5 这一步的意义

这是项目第一次让环境**真正动态**，且 PPO 应对了。之前 DQN 在静态环境都学不深，更别说动态。PPO 在"不可见的动态风险"下稳定学到"打了就跑"——这是真实渗透的核心行为之一。

这也验证了 on-policy 算法对动态环境的适应性：检测率在变，但变化是可预测的规则（不是博弈），PPO 用当前策略数据持续适应，没有 DQN 的遗忘问题。

### 11.6 动态检测的两个层级

本节实现的是**层级一（规则式）**：检测率按预设规则变化，没有防御智能体在决策。还有**层级二（防御智能体/蓝队）**：一个独立 RL 智能体决定在哪里部署检测、隔离哪个节点，攻防双方同时学习，形成非平稳博弈。

层级二是真实攻防对抗的终极形态，但工程量是论文级（双方动作空间、奖励、自博弈、非平稳收敛）。本项目的策略是先把层级一做扎实——它是层级二的地基（动态检测机制已就位，将来让规则变成可学习决策即可）。直接跳层级二风险过高。

## 12. 当前状态、已知局限与改进方向

### 12.1 当前成果 (已固化)

智能体已能**自主走通完整攻击链**并拿到 flag。一次成功的演示 (日志 `rl-agent-v2-21024.out`)：

```
0  MOVE_2    Gateway → JumpBox
1  RECON     JumpBox
2  VULN_SCAN JumpBox
3  EXPLOIT_CREDS  ✅ 攻陷
4  PRIVESC   JumpBox
5  MOVE_3    JumpBox → DC        ← 关键决策：选了未访问的 DC，而非回头
6  RECON     DC
7  VULN_SCAN DC
8  EXPLOIT_CVE    ✅ 攻陷
9  PRIVESC   DC
10 EXFIL     DC  🏁 FLAG!
总奖励 +38.70 | 检测 1 次
```

智能体自主学会了三件事：
- **选安全入口**：在 Gateway 选 JumpBox (检测 0.15) 而非 WebServer (0.70)
- **完整攻击链**：每个节点都走 RECON→VULN_SCAN→EXPLOIT→PRIVESC，不跳步
- **不回头**：JumpBox 攻陷后直奔 DC，不在节点间振荡

**成果固化方式**：DQN 版用 `models/best_v2.pt`（best 快照，见第 10 章关于 best 的讨论）；PPO 版用 `models/best_ppo.pt`（final 即 best，on-policy 不遗忘）。可用 `python src/demo_v2.py` / `python src/demo_ppo.py` 直接加载模型重跑，无需重训。

### 12.2 已知局限

1. **DQN 学不深**（已换 PPO 解决，详见第 10 章）：DQN 因经验回放灾难性遗忘，后期策略持续退化，演示靠 best 快照碰运气。PPO 已治本。
2. **参数未经真实数据拟合**：检测率、动态 α、利用率等参数目前是经验估计（见 12.4 参数依据）。敏感度分析（12.5）证明参数 ±20% 波动下策略 robust，缓解了这个问题，但真实数据拟合仍是长期目标。
3. **部分可观测（进行中）**：已隐藏"被检测次数"（智能体不可见 IDS 告警计数，只从惩罚奖励间接推断）。但未访问节点的"存在性"仍是已知的（state 给了固定节点槽位），且智能体只看当前观测、无历史记忆。完整的 POMDP 还需 RNN 策略追踪信念状态。
4. **单智能体**：没有蓝队 (防守方) 的对抗，环境是规则式动态（检测率按预设规则升降），不是博弈。真实攻防是非平稳博弈。
5. **离散动作**：固定动作数。真实工具是参数化的 (选 CVE + 选 payload + 选时机)。
6. **PPO 更新时未严格 mask**：采样时严格 mask（非法动作绝不执行），但梯度更新时依赖奖励塑形让非法动作自然低概率，未逐样本重建 mask。4 节点下验证有效，若更大环境训练不稳再补严格 mask。

### 12.4 参数依据与局限

环境里的检测率、动态 α、利用率等参数目前是**经验估计**，未经真实数据拟合。逐项说明依据与局限：

| 参数 | 当前值 | 依据 | 局限 |
|------|--------|------|------|
| 节点检测率 (base) | 0.05/0.15/0.70 | 经验：暴露在 DMZ 的 Web 服务 IDS 规则覆盖全，类比 CIC-IDS 中 HTTP 攻击检出率 | 非真实数据拟合，依赖具体 IDS 产品规则 |
| α_attack (攻击升级) | 0.08 | 经验：每次攻击性动作使 SOC 警觉度上升约 8% | 无公开时序数据支撑 |
| α_detected (确认升级) | 0.15 | 经验：确认入侵后升级比单纯告警更快 | 同上 |
| α_decay (衰减) | 0.01 | 经验：假设 SOC 警觉半衰期约一个 episode 周期 | 真实 SOC 响应是离散事件非平滑衰减 |
| 利用成功率 cve | 0.70 | CVSS exploitability 经验映射 | 实际成功率依赖目标环境 (补丁/ASLR) |
| 利用成功率 creds | 0.85 | 弱口令字典命中概率较高 | 依赖口令策略 |
| 利用成功率 sqli | 0.60 | SQL 注入依赖注入点与过滤 | 依赖 WAF 配置 |

**局限的诚实评估**：这些参数是"有依据的估计"，不是"测量值"。真实数据拟合需要 CIC-IDS2017 等数据集（能拟合静态检测率）+ 红队实测（拟合动态 α 和利用率），工程量大且数据获取难。本项目选择"经验估计 + 敏感度分析"的务实路径——先用敏感度分析证明参数不精确问题可控，真实拟合作为长期目标。

### 12.5 敏感度分析：参数不精确怎么办

回答的问题：**如果真实环境的参数和我们设的有偏差，学到的策略还有效吗？**

实验（`src/sensitivity.py`，日志 21295）：以 `env_default_dynamic_prob.yaml` 为基线，逐个扰动单个参数组，每组 3 个种子训练 PPO 60 rollout、评估 100 episode，取中位数。

| 实验 | reward | flag% | det | steps | 3种子 flag% |
|------|--------|-------|-----|-------|------------|
| det×0.8 (IDS 弱) | +28.49 | 100% | 2.75 | 11.6 | [100/100/100] |
| det×1.0 (基线) | +27.09 | 100% | 3.01 | 11.7 | [100/100/100] |
| det×1.2 (IDS 强) | +25.69 | 100% | 3.32 | 11.6 | [100/100/100] |
| cve=0.5 (利用难) | +19.47 | 100% | 3.16 | 11.7 | [100/100/100] |
| cve=0.7 (基线) | +27.09 | 100% | 3.01 | 11.7 | [100/100/100] |
| cve=0.9 (利用易) | +28.98 | 100% | 2.87 | 11.2 | [100/100/100] |
| α=0.04 (动态弱) | +27.83 | 100% | 2.88 | 11.6 | [100/100/100] |
| α=0.08 (基线) | +27.09 | 100% | 3.01 | 11.7 | [100/100/100] |
| α=0.12 (动态强) | +26.06 | 100% | 3.23 | 11.6 | [100/100/100] |

**结论：9 组参数扰动 × 3 种子 = 27 次训练，全部 100% 拿到 flag。策略对参数波动 ROBUST。**

观察到的合理趋势（说明模型行为正确，不是"瞎学"）：
- 检测率上升（×0.8→×1.2）：检测次数从 2.75 升到 3.32，奖励从 +28.5 降到 +25.7——IDS 越强，被检测越多、奖励越低，但策略仍能拿 flag
- 利用率上升（0.5→0.9）：奖励从 +19.5 升到 +29.0——利用越容易成功，奖励越高（低利用率时偶有重试消耗步数）
- 动态 α 上升（0.04→0.12）：检测次数从 2.88 升到 3.23——告警升级越快，被检测越多

**这回答了"真实参数会变化怎么办"**：在我们手编参数 ±20% 波动范围内，PPO 策略不受影响。即使真实环境的检测率/利用率和设定值有偏差，学到的攻击策略仍然有效。参数不精确问题在敏感度分析覆盖的范围内可接受。

**边界与诚实声明**：敏感度分析只覆盖了 ±20% 检测率、利用率 0.5-0.9、α 0.04-0.12 的范围。超出这个范围（如检测率翻倍、利用率极低）策略是否仍 robust 未验证。且分析在 4 节点环境，更大规模未做。真实数据拟合仍是最终保证。

### 12.6 真实数据拟合：CIC-IDS2017 检测率

敏感度分析证明了"参数 ±20% 波动策略 robust"，但参数本身仍是经验估计 (12.4)。这一步把**检测率从经验估计升级为真实数据拟合**，是项目从"仿真"走向"有数据支撑"的关键。

**数据集**：CIC-IDS2017（UNB 加拿大网络安全研究所，2017）。在受控网络用真实工具发起各类攻击，用 CICFlowMeter 提取每个网络流的 78 个统计特征 + 攻击标签。我们用的是整理过的 parquet 版本（按攻击类型分文件）。

**拟合方法**（`src/fit_detection_from_cicids.py`）：
1. 对每类攻击，用逻辑回归二分类（攻击 vs Benign）算 AUC——AUC 衡量"流量可识别度"
2. 按攻击类型映射到节点：SSH-Patator→JumpBox、Web Attacks→WebServer、Infiltration→DC
3. 按 AUC 相对排序分三档映射 detection（0.15/0.45/0.70）

**关键设计决策与诚实声明**：
- **不直接用 AUC 绝对值**。离线用全部 78 特征做分类，AUC 都接近 1.0（完美可分），直接映射会让所有节点 detection 飙到 0.89，失去"安全 vs 危险"的决策张力。这只反映"事后拿全部特征分类"的能力，不等于真实 IDS 实时检测率（在线检测、特征有限、误报约束）。
- **用 AUC 的相对排序 + 经验区间**。保留数据的核心信息（哪类攻击更易识别），但绝对值用经验校准的三档。AUC 是"流量可识别度代理指标"，非真实 IDS 检测率——CIC-IDS 的 Label 是真实攻击标注（ground truth），不是 IDS 检测输出。
- **仅检测率数据驱动**。动态 α 和利用率仍经验估计（CIC-IDS 无法拟合这些）。

**拟合结果——纠正了经验直觉**：

| 节点 | 对应攻击 | AUC | 拟合 detection | 经验版 detection |
|------|---------|-----|---------------|-----------------|
| JumpBox | SSH-Patator | 0.996 | 0.70（最易检测） | 0.15（最安全）|
| WebServer | Web Attacks | 0.993 | 0.45 | 0.70 |
| DC | Infiltration | 0.965 | 0.15（最隐蔽） | 0.70 |

**两个反直觉但符合数据的发现**：
1. **SSH 爆破流量最易检测**。我们凭直觉把 JumpBox（SSH 跳板）设成低检测"安全入口"，但 CIC-IDS 显示 SSH-Patator 的 AUC 最高（0.996）——爆破行为在流量上极度显著（大量失败连接、短时间高频尝试），真实 IDS 抓 SSH 爆破很在行。**数据纠正了"SSH 跳板机低调安全"的手编假设**。
2. **内网渗透最难检测**。DC（Infiltration）AUC 最低且仅 36 个样本——内网渗透流量稀少且隐蔽，IDS 难抓。flag 节点反而最不易被检测，符合"渗透进来后横向移动低慢隐秘"的现实。

**拟合版 PPO 训练**（`configs/env_fitted.yaml`，日志 21562）：avg +30、flag% 100%，稳定收敛。**演示策略翻转**——PPO 选择了 WebServer 入口（detection 0.45）而非 JumpBox（0.70），走 Gateway→WebServer→DC 路径。这是数据驱动的最优路径，和经验版（Gateway→JumpBox→DC）完全相反。演示中还出现了概率利用重试（EXPLOIT_CVE 失败 2 次后第 3 次成功），直观证明 PPO 学会了"失败重试"。

**这一步的意义**：检测率不再靠拍脑袋，而是有 CIC-IDS2017 数据支撑。拟合结果纠正了经验直觉，PPO 在数据驱动的 detection 下学到了符合数据的最优路径。项目首次具备"数据→参数→策略"的完整链路。局限是：仅检测率数据驱动（动态 α、利用率仍经验估计），且 AUC 是代理指标非真实检测率。

### 12.7 真实 IDS 验证：代理指标的局限

12.6 用 AUC（流量可识别度）作为检测率代理，拟合出 detection 值。但 AUC 是离线分类能力，不等于真实 IDS 在线检测能力。这一步用真实 Suricata + 社区规则回放 CIC-IDS2017 PCAP，验证代理指标到底有多可信。

**实验**（Suricata 4.1.2 + Snort community rules 4013 条启用，回放原始 PCAP）：

| 节点 | 攻击 | AUC 代理 | 真实 IDS 检出 | 真实攻击流量 |
|------|------|---------|-------------|------------|
| JumpBox | SSH 爆破 | 0.996 → det 0.70 | **0 告警** | SSH-Patator 流量存在 |
| WebServer | Web 攻击 | 0.993 → det 0.45 | **0 告警** | 72 个 SQLi 特征 HTTP 请求 |
| DC | Infiltration | 0.965 → det 0.15 | 仅 5 条 cmd.exe + 3 条 dir listing（间接迹象） | — |

**核心发现：AUC 代理系统性严重高估真实 IDS 检测能力。**

三个节点，代理都给到 0.96-0.99 的 AUC（"流量高度可分"），但真实 Suricata + 社区规则对对应的攻击**几乎全 0 检出**。这揭示了代理指标的致命局限：

1. **AUC 衡量的是"离线可分性"，不是"在线可检测性"**。用 78 个流量特征做逻辑回归分类，能完美区分攻击 vs 正常（AUC 0.99），但真实 IDS 在线检测靠的是**规则匹配**，不是机器学习分类。流量"可分"不等于"有规则能检"。

2. **真实检测能力 = 规则覆盖，不是流量特征**。社区规则没有针对这些攻击的具体规则：SSH 爆破加密（IDS 看不到认证失败），SQLi 的具体 payload 没规则匹配。要检出得用订阅规则（Snort/ET 订阅版覆盖最新攻击）或主机日志（看认证失败）。

3. **加密协议尤其失真**。SSH 爆破在流量上只有正常握手（client JSCH ↔ server OpenSSH），AUC 高是因为用了离线统计特征，但真实 IDS 从加密流量看不出任何"爆破"特征。代理对加密流量系统性高估。

**这一步的价值（负面结果也是结论）**：

- **回答了最初疑虑的深层版本**。不只"真实参数会变化"，而是"我们用的代理指标本质上和真实检测是两回事"。AUC 0.996 不代表真实 IDS 能 99.6% 检测——它只代表"事后拿全部特征能区分这两类流量"。
- **敏感度分析的意义反而被强化**。策略对 detection ±20% 波动 robust，正是因为策略不该依赖 detection 的精确值——真实检测能力本就难以精确刻画，robust 策略才是务实目标。
- **定义了"数据驱动"的诚实边界**。项目能做"数据驱动的代理 detection"，但做不到"真实 IDS detection"——后者需要订阅规则或主机日志，超出当前条件。这个边界值得明确，避免过度宣称"数据驱动"。

**没做的事与原因**：B1 没能用真实检出率替换 detection 值——结果全是 0，设 0 会让环境失去决策张力（无检测则无规避决策）。因此 12.6 的 AUC 拟合值保留，但**降级理解为"数据驱动的代理 detection"**，而非真实 IDS 检测率。要得到真实 detection，需：(1) Snort/ET 订阅规则覆盖 CIC-IDS 攻击，或 (2) 主机日志（SSH 认证失败、Web 错误响应）作为检测源，而非网络 IDS。两者都超出本项目条件，留作未来工作。

### 12.8 蓝队对抗：self-play 的边界探索

前面的所有工作都是单智能体（红队攻击）。最后一步尝试双智能体——引入蓝队防御智能体，通过 self-play（交替训练 + 对手快照池）让双方同时学习，形成攻防博弈。这是这个方向风险最高的一步，结果也最不确定。

**设计**（四层兜底验证）：
- 蓝队动作：MONITOR / HARDEN（加固节点，提升检测率）/ ISOLATE（隔离节点，拉到检测率上限，不切断路径）
- 蓝队观测：告警计数 + 探测痕迹 + 自身防御状态 + 检测率（不看红队位置，不开上帝视角）
- 双方奖励：非零和，蓝队"隔离正确 +8、误隔离 -3、正确加固 +5"（鼓励精准防御）
- 蓝队预算：每 episode 最多 3 次防御动作（防无限隔离）
- 训练机制：self-play 交替训练 + 对手快照池（池大小 5，随机抽历史对手防策略循环）
- 背景误报：每步 8% 概率随机节点 +1 告警（模拟 IDS 误报，避免蓝队靠告警精准定位红队）

**兜底层 1（随机蓝队）**：红队 vs 随机防御的蓝队。红队 flag% 稳定在 55-62%，验证红队在对抗环境下能学。

**兜底层 2（规则蓝队）**：红队 vs 规则蓝队（隔离告警最多节点）。修复前蓝队过强（红队 flag% 10-14%，卡死），修复后红队 100%，验证环境机制工作。

**self-play（三轮实验）**：红蓝双方同时 PPO 学习，交替训练。三轮均以蓝队学不动、红队 100% 结束。

| 实验 | 蓝队观测 | 红队后10轮flag% | 蓝队行为 |
|------|---------|--------------|---------|
| 原始奖励（误隔离-8/正确+4） | 18D（告警为主） | 稳定 100% | 收敛到不防御 |
| 调奖励（误隔离-3/正确+8） | 18D | 稳定 100% | 仍收敛到不防御 |
| 加探测痕迹（预测信号） | 22D（告警+探测痕迹） | 稳定 100% | 仍收敛到不防御 |

所有三轮实验中，前 10 round 红队 flag% 在 20-100% 间剧烈震荡，后 10 round 稳定在 100%、蓝队 block% 稳定在 0%。震荡来自快照池随机抽对手的噪声，不是蓝队在学。

**根因分析**：蓝队学不动不是奖励设计问题（调过没用），也不是观测不够（加过预测信号没用），而是 **self-play 的非平稳性**。蓝队在 self-play 里面对的对手（红队）每轮在变——这一轮学到"对付红队快照 A 的策略"，下一轮对手变成红队快照 B，策略失效。梯度互相抵消，蓝队最终收敛到"不防御"——因为主动防御在不同对手上的收益方差太大，远不如"不动"稳定。蓝队永远在追一个移动靶。

**结论**：在当前框架（self-play + 不完美信息 + 非零和）和资源条件下，蓝队学不到有意义的主动防御策略。这是一个诚实的负面发现，和 12.7 的"代理指标局限"性质一致——**它定义了当前方法论的边界**，而非"做不出来是能力问题"。

**要跨越这个边界需解决的是**：非平稳博弈的收敛性（需虚拟博弈/种群训练/NFSP 等专门博弈算法）或蓝队观测升级（给预测性而非仅滞后信号）。两者都超出当前条件——前者是论文级算法，后者滑向给蓝队开上帝视角。

代码保留在 `src/multiagent/`（隔离于单智能体基线），兜底层 1/2 均可复现。self-play 的三轮实验数据可作为"RL 攻防博弈在当前框架下有明确天花板"的实证。
